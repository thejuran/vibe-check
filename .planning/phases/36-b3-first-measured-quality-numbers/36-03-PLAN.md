Score every isolated run against the PROVABLY pre-registered answer key and write the first aggregate
catch-rate / FP-rate report (B3-02 scoring half + B3-03). Scoring is GATED on the key's pre-registration
proof read from the committed manifest blob at MANIFEST_COMMIT — derived, not trusted (codex pass-3 fix 1 + pass-4 fix 1: the manifest's own ordering is proven; a post-run manifest edit or a manifest+runs same-commit landing EXITS NON-ZERO) — and reads the key ONLY from
the committed blob at ANSWER_KEY_COMMIT (`git show`), never from the live working file: a post-run edit to
the live key could otherwise change the SITE/AXIS/BAND rows actually used. The gate EXITS NON-ZERO on
digest mismatch, proves ancestry (`git merge-base --is-ancestor ANSWER_KEY_COMMIT HEAD`), hard-fails unless
`git status --porcelain runs/` is EMPTY and every runs/ commit descends from the key commit AND from MANIFEST_COMMIT (codex pass-3
fix 2 + pass-4 fix 1), and verifies each diff's archived FULL-worktree tree.diff sha256 (`git diff` with no pathspec — pass-5
fix 1) is identical across its 3 runs and equals the kit-build EXPECTED_TREE_DIFF_SHA256 (codex pass-3
fix 3). Scoring HARD-STOPS (exit non-zero) before aggregation if any expected run is unscoreable — every
diff needs exactly 3 scoreable runs; only an explicit owner-authored waiver recorded visibly in the report
can change the denominator (codex pass-5 fix 3, enforcing D-06). Scoring reads the machine-readable
`state.passes[-1].findings[]` only (D-06 — never a transcript) and only for runs whose state carries
`len(passes)==1` AND `head_sha == the key row's base_sha` (a fresh isolated sample of the pinned tree),
applies the D-07 three-gate catch rule and the D-08 FP rule, aggregates per D-09 (no rounding), and APPENDS
the report to RESULTS-v2.9.md with honest limitations, a plain-language owner summary, and the D-11
proceed/don't/need-more-data verdict.

Purpose: This closes the self-documented "no aggregate numbers" gap (RESULTS.md:64) — the first
catch-rate/FP-rate the tool has ever had — and produces the explicit input to next milestone's scoping
of the B3-gated design challenges. The score-from-blob + ancestry + runs-descent + tree.diff gate makes the
number PROVABLE end to end: the key that scored the runs is byte-identical to the one committed before any
run existed, and every scored run reviewed exactly the planted diff.

Output: `SCORING-b3.md` (the auditable per-run worksheet, headed by the full gate results) + an
appended B3 section in RESULTS-v2.9.md.
